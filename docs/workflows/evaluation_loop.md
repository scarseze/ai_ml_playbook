# 🔄 Evaluation & A/B Testing Loop (Continuous Monitoring)

В AI-разработке тестирование является непрерывным процессом, так как данные (Data Drift) и поведение пользователей меняются с течением времени.

## 1. Автоматическое тестирование при PR (Pull Request)
- Любое изменение промпта, конфигурации RAG или весов модели должно описываться `manifest.json`.
- `MLOps Engineer` настраивает CI-пайплайн внутри SGR-Kernel.
- Pipeline разворачивает Sandboxed-контейнер и прогоняет Golden Dataset (эталонный).
- `LLM Evaluator` анализирует результаты и генерирует отчет (Experiment Report).

## 2. Трекинг Экспериментов (Experiment Tracking)
- Все пайплайны используют MLflow (или встроенный логгер SGR-Kernel).
- Каждому запуску присваивается уникальный Run ID, связывающий `manifest.json`, хеш датасета, промпт и логи (WAL).

## 3. Производственный мониторинг (Continuous Evaluation)
- Периодический срез случайных ответов системы в Production (с PII очисткой).
- Прогон через "модель-СУДЬЮ" (LLM-as-a-judge).
- Отслеживание "Confidence Score" – готовности модели ответить правильно.
- Если средний Confidence Score за час падает — генерируется алерт (PagerDuty) для MLOps и Evaluator.

## 4. A/B Тестирование (A/B Test)
- `AI Product Owner` может предложить пустить `10%` трафика на новую модель или промпт.
- `MLOps Engineer` настраивает маршрутизацию (Routing) в **SGR-Kernel Routing Node**. Это достигается путем создания динамического правила в оркестраторе:
  - SGR-Kernel анализирует заголовки входящих запросов (например, наличие хеша пользователя).
  - Направляет вызов на специфическую версию агента (Скилла), подменяя манифест "на лету" (Control vs Test manifest).
- Сравнение двух групп (Control / Test) по ключевым бизнес-метрикам (CTR, Engagement) и метрикам качества ответов.
- После достижения статистической значимости принимается решение о раскатке на `100%`.
