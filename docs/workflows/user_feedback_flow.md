# 🔁 User Feedback Flow (RLHF)

Процесс сбора, классификации и использования обратной связи пользователей для дообучения моделей и обновления промптов. Модель не должна стагнировать.

## 1. Сбор данных (Telemetry Collection)
- Интерфейсы собирают маркеры обратной связи (Thumbs up / Thumbs down).
- Сохраняются контекст, текст запроса, сгенерированный ответ, Trace ID, и (опционально) текстовое примечание пользователя.
- MLOps настраивает отправку данного лога в отдельное хранилище SGR-Kernel (Feedback Store), где Data Engineer применяет PII фильтрацию.

## 2. Классификация и разметка (Data Engineer / Evaluator)
Данные агрегируются еженедельно. LLM Evaluator (или автоматическая "Модель-Судья") классифицирует дизлайки:
- (A) Фактологическая ошибка RAG (Не нашел документ).
- (B) Форматирование (Вывел не то, что просили).
- (C) Тональность (Грубо, неэмпатично).
- (D) Галлюцинация (Выдуманный факт).

## 3. Обработка Фидбека
В зависимости от классификации:
- **Для (A) RAG:** `Data Engineer` проверяет полноту базы. `RAG Specialist` тюнит Retrieval.
- **Для (B, C) Ошибки стиля/формата:** `RAG Specialist` обновляет Prompt.
- **Для (D) Галлюцинации:** Формируется датасет (Preference Dataset). Пользовательский запрос размечается: `Chosen response` (Правильный ответ написанный Аналитиком) vs `Rejected response` (Ответ модели). Это формирует базу для RLHF/DPO.

## 4. Fine-Tuning Цикл
- Накопленный датасет (например 10k "плохих" и "хороших" ответов) инициирует `finetuning_flow.md` под руководством AI Product Owner.

## Метрика Успеха (Success Metric)
- Снижение процента дизлайков (Thumbs down rate) на N% в течение месяца после обновления модели/промпта.
