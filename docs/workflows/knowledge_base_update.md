# 📚 Knowledge Base Ops (Поддержка Векторной Базы)

Векторная база (Knowledge Base) — это "память" нашей RAG-системы. Документация в компании постоянно меняется, поэтому RAG не может быть настроен один раз и забыт. Нам нужен процесс регулярного обновления и переиндексации (Knowledge Base Ops).

## 1. Регламент Обновления (Как часто?)
- **Hot Updates (Ежедневные)**: Инкрементальное добавление или удаление небольших статей (News, Wiki). Выполняется автоматически по webhook-событию из системы документации (Confluence/Notion/Git).
- **Cold Indexing (Еженедельные/Ежемесячные)**: Полная перестройка всей базы (Reindexing). Выполняется по расписанию (CronJob) внутри SGR-Kernel.

## 2. Кто триггерит процесс?
- Автоматизированные триггеры настраиваются **Data Engineer**.
- Ручной запуск переиндексации (при смене модели эмбеддингов, например, с `text-embedding-ada` на `text-embedding-3`) инициирует **RAG Specialist**.

## 3. Валидация новых эмбеддингов
Перед тем как новая векторная база (или новая коллекция) перейдет в Production, она должна быть проверена:
1. **Shadow Testing**: Новая коллекция в Milvus/Qdrant поднимается параллельно основной (Staging Collection).
2. **Evaluation Pipeline**: `LLM Evaluator` запускает Unit-тесты (Golden Dataset) поверх новой базы. Если показатель `Context Relevance` проседает — обновление блокируется.
3. Процесс смены баз контролируется через `manifest.json` в SGR-Kernel (указание версии коллекции базы данных).

## 4. Очистка устаревших данных (TTL & Pruning)
Data Engineer обязан:
- Реализовать версионирование документов (метадата: `updated_at`).
- При переиндексации "мягко" удалять устаревшие чанки (Soft delete) и гарантировать, что LLM не выдаст клиенту правила возврата товара за 2015 год.
